Overview

매번 팀원이 부족해 경기가 취소되던 경험에서 출발한 프로젝트입니다. 위치 기반으로 가까운 팀을 매칭하고, 경기 일정부터 기록까지 한 곳에서 관리합니다.

백엔드와 프론트엔드를 모두 혼자 설계·구현하는 것을 목표로 잡았습니다. 실무에서는 주로 Java/Spring 기반으로 작업했지만, 이 프로젝트를 통해 Next.js와 모던 프론트엔드 생태계를 깊게 익혀보고자 했습니다.

Tech Stack

Spring Boot 3 MyBatis MySQL Next.js 14 TypeScript Naver Maps API GitHub Actions AWS EC2

Key Decisions

  • MyBatis XML 전용 규칙. 어노테이션 SQL 금지. 모든 쿼리를 XML로 분리해 리뷰 가능성과 유지보수성을 우선시했습니다.
  • 네이버 지도 API 선택. 국내 사용자 대상이기 때문에 구글 대비 비용 효율과 한국 주소 정확도 모두 우위였습니다.
  • EC2 단일 인스턴스 시작. ECS Fargate도 검토했지만, 초기 단계에는 비용과 디버깅 용이성 때문에 EC2로 출발.
  • Next.js Static Export. 공개 페이지 대부분이 정적이므로 SSR 대신 SG로 단순화.

Architecture Snippet

# docker-compose.dev.yml
services:
  db:
    image: mysql:8.0
    environment:
      MYSQL_DATABASE: moch
      MYSQL_ROOT_PASSWORD: ${DB_PASSWORD}
    volumes:
      - mysql-data:/var/lib/mysql
    ports: ["3306:3306"]

  api:
    build: ./backend
    environment:
      SPRING_PROFILES_ACTIVE: dev
      NAVER_CLIENT_ID: ${NAVER_CLIENT_ID}
    depends_on: [db]
    ports: ["8080:8080"]

Lessons Learned

"실무와 다른 환경에서 처음부터 끝까지 설계하는 것"의 가치를 다시 느꼈습니다. 특히 CI/CD와 프론트엔드를 직접 다뤄보니, 평소 API 설계할 때 프론트가 겪는 불편함이 더 선명하게 보입니다.
