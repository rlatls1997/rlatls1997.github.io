Problem

수십 개의 쇼핑 콘텐츠 소스에서 매 순간 데이터가 들어옵니다. 어느 날 갑자기 추출량이 평소의 10분의 1로 떨어져도 알아차리지 못한다면? 수집 파이프라인의 침묵은 곧 사고입니다.

단순한 임계값 기반 알람은 이미 있었지만, 소스마다 볼륨이 제각각이라 고정값으로는 제대로 동작하지 않았습니다. 소스별로 자기 자신의 과거와 비교하는 통계적 접근이 필요했습니다.

Approach

Z-score 기반 이상 감지에 "하드 스톱" 보조 조건을 결합한 하이브리드 방식을 설계했습니다.

threshold = μ + MAX(K · σ, μ · R)
μ = moving average · σ = std deviation · K, R = tunable
  • Z-score 파트 (K·σ). 정상 분포 내에서의 튀는 구간을 잡습니다.
  • 비율 하한 (μ·R). σ가 너무 작게 계산되는 초기 구간에서 오탐을 줄입니다.
  • MAX 조합. 둘 중 엄격한 쪽을 채택해 "이건 진짜다" 싶은 경우만 알람.

Architecture

// Redis ZSET 기반 슬라이딩 윈도우
public void record(String sourceId, long count) {
  String key = "extract:win:" + sourceId;
  long now = System.currentTimeMillis();

  redis.opsForZSet().add(key, now + ":" + count, now);
  redis.opsForZSet().removeRangeByScore(
    key, 0, now - WINDOW_MS   // 1시간 윈도우
  );
  redis.expire(key, Duration.ofHours(2));
}

public boolean isAnomaly(String sourceId, long current) {
  Stats s = computeStats(sourceId);   // μ, σ from ZSET
  double threshold = s.mu - Math.max(
    K * s.sigma,
    s.mu * R
  );
  return current < threshold;
}

감지되면 Naver Works Bot을 통해 담당자 채널로 즉시 알람. 복구 여부까지 추적해 "복구됨" 메시지도 자동 전송하게 만들었습니다.
