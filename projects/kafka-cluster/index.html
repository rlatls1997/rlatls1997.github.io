Problem

모든 WAS 노드가 같은 이벤트를 받아서 각자의 로컬 캐시를 갱신해야 하는 요구사항. 일반적인 컨슈머 그룹 구조로는 해결되지 않는 문제였습니다.

Kafka의 컨슈머 그룹은 그룹 단위로 메시지를 나눠 가져가는 구조입니다. 같은 그룹 안의 노드들이 파티션을 분배받기 때문에, 4개 노드가 같은 그룹을 쓰면 각자 다른 메시지만 받습니다. 캐시 갱신 같은 브로드캐스트 목적에는 부적합합니다.

Design Comparison

— Naive

모든 노드가 cache-updater 같은 동일 그룹 사용.
→ 각 노드가 일부 메시지만 받음.
→ 캐시 일관성 깨짐.

— Adopted

각 노드가 cache-updater-${HOSTNAME}로 고유 그룹 할당.
→ 모든 노드가 모든 메시지 수신.
→ 완전한 브로드캐스트.

Configuration

// 노드별 고유 그룹 ID 주입
@Configuration
public class KafkaConsumerConfig {

  @Value("${HOSTNAME:unknown}")
  private String hostname;

  @Bean
  public ConsumerFactory<String, String> cacheConsumerFactory() {
    Map<String, Object> props = new HashMap<>();
    props.put(GROUP_ID_CONFIG, "cache-updater-" + hostname);
    props.put(AUTO_OFFSET_RESET_CONFIG, "latest");
    props.put(ENABLE_AUTO_COMMIT_CONFIG, false);
    return new DefaultKafkaConsumerFactory<>(props);
  }
}

Considerations

  • Offset reset = latest. 새로 뜨는 노드는 과거 캐시 이벤트를 재생할 필요 없음. 시작 시점 이후만 처리.
  • Auto-commit = false. 캐시 갱신 실패 시 재시도 가능성을 열어두기 위해 수동 커밋.
  • 그룹 정리. 노드 교체 시 과거 그룹이 남는 문제는 Kafka의 offsets retention(기본 7일)로 자연 정리.
  • 모니터링. 그룹 수가 노드 수만큼 늘어나므로 컨슈머 랙 대시보드에서 필터링 필수.
